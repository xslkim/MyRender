右半边代表 CPU：一块揭开了顶盖的银色处理器芯片，内部电路纹理清晰、泛着明亮的科技蓝光，几个悬浮的发光圆球像调试断点一样停在电路上空，一束放大镜般的探查光打在芯片上，氛围通透、开放、尽在掌控。
左暗右亮、左封闭右敞开，对比强烈。背景接近 #0d1117 的深蓝黑，点缀色科技蓝 (#58a6ff)，3D 渲染质感，干净、有景深、轻微辉光；画面中不要出现任何文字或字母；主体居中偏上，画面最底部保留一条干净的深色空间方便叠加字幕。

--- narration ---
学 Unity 的 Shader 时
最难受的一点是
它跑在 **GPU** 上
你没办法设断点
没办法打印一个变量的值
出了问题只能靠猜
那不如把整条管线
搬到 **CPU** 上用 C++ 重写一遍
慢一点完全没关系
但每一个像素、每一个中间变量
你都能单步跟踪、随便打印
甚至直接截图看出来

--- narration ---
不管是什么显卡
真实的渲染都是这样一条流水线
顶点数据先进来
顶点着色器做坐标变换
裁剪掉看不见的部分
光栅化把三角形铺成像素
片元着色器算出每个像素的颜色
最后写进帧缓冲显示出来
GPU 用硬件做这些
我们就用 C++ 把每一步老老实实模拟一遍
这条流水线
就是后面两集的主线

--- narration ---
整个工程就三大块
第一块是 **core**，场景层
用 JSON 描述相机、光源和物体
第二块是 **Render**，渲染管线
负责变换、裁剪、光栅化
第三块是 **gpu**，着色层
我们故意让它写得像 Unity 的 HLSL
连函数名都对得上
这样你在 Unity 里学到的东西
能一对一搬过来读

  "4  逐三角形：顶点着色 → 裁剪 → 光栅化"
  "5  颜色缓冲转 sRGB → 交给屏幕"
右侧占画布 40%，展示一段精简的 Render() 调用伪代码，等宽字体 26px，语法高亮（关键字 #ff7b72，函数 #d2a8ff，注释 #8b949e）：
  FrameStart(cam, light);
  for (obj : objects) {
    UpdateModelMatrix(obj);
    Draw(obj.mesh, obj.mat);
  }
  // 颜色缓冲 → sRGB → 屏幕
[0s] 右侧代码淡入。
[1s] 左侧五个节点从上往下依次滑入并高亮，间隔 0.6s，高亮节点与右侧对应代码行用 accent 连线呼应。

--- narration ---
先建立一个全局的心理模型
程序每画一帧
其实只做五件事
第一步，清空颜色缓冲和深度缓冲
第二步，把相机和光源上传到一组全局变量
第三步，遍历每个物体，算出它的模型矩阵，再调用 Draw
第四步，对每个三角形，做顶点着色、裁剪、光栅化
第五步，把算好的颜色转成 sRGB，交给屏幕
这五步
就是这一集的地图
其中第三步和第四步
是后面两集的重头戏
